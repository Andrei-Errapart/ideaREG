## Idea

### Title

**Virtual SD I/O** — using emulated SD card in FPGA fabric as a high-bandwidth data path between hard processor and FPGA fabric, accessible through standard filesystem operations.

### Description

The idea applies to FPGA SoC devices that have a hard SD card controller block whose pins can be routed either to external device pins (for use with physical SD cards) or into the FPGA programmable logic fabric (for use with arbitrary logic). The AMD Zynq 7000 family is one example; other FPGA SoC families with similar architectural flexibility apply equally.

Instead of routing the SD controller to physical SD card pins, the controller is routed into the FPGA fabric. Inside the fabric, an SD card emulator is implemented that responds to the SD protocol as a real card would. The host system, running its standard operating system (typically Linux), sees what appears to be a normal SD card and interacts with it through standard filesystem operations.

What sits behind the emulated card is the design's choice. Three useful configurations:

**Configuration 1 — BRAM-backed storage.** The emulated card's storage is implemented in FPGA block RAM. The host can read and write files to this storage using standard filesystem commands, with the data persisting only as long as the FPGA configuration remains active. Useful for configuration data, parameters, or small datasets that need to be accessible from the host but generated or consumed by fabric logic.

**Configuration 2 — External memory backed storage.** The emulated card's storage is implemented in DDR or other external memory accessible from the fabric. Larger capacity than BRAM-backed configurations. Useful when the data sizes exceed what BRAM can accommodate.

**Configuration 3 — Streaming interface backed.** The emulated card does not actually store data persistently. Instead, file operations on the host side are translated into AXI streaming transactions on the fabric side. When the host writes a file, the data flows out through an AXI stream to fabric logic that consumes it. When the host reads a file, the data is supplied by fabric logic through an AXI stream that the host receives as if reading from storage.

Configuration 3 is the most powerful of the three. It provides a high-bandwidth bidirectional data path between the host processor and the fabric, with the host using standard Linux filesystem APIs and the fabric using standard AXI streaming. Neither side needs to know about the unusual bridging happening inside the SD card emulator.

### Why this idea matters

Moving large amounts of data between the hard processor and the FPGA fabric in a Zynq-class device is a common but non-trivial engineering problem. The standard mechanisms — AXI memory-mapped transactions through the GP or HP ports, AXI streaming through DMA engines configured by the host — all work but require coordinated programming on both sides. Bandwidth, latency, and software complexity all need to be managed carefully.

Virtual SD I/O provides an alternative path that uses interfaces both sides already know:
- The host side uses standard filesystem operations (open, read, write, close on files in a mounted filesystem). Any Linux programmer can use this interface without learning anything new.
- The fabric side uses standard AXI streaming, the same interface that countless IP cores already implement.

The unusual work — translating between SD card protocol on one side and AXI streaming on the other — is encapsulated in the fabric-side SD card emulator. It is written once and reused across projects.

### Domain or field

FPGA SoC design, Zynq architecture, host-fabric communication, embedded systems, AXI interfacing.

### References to related ideas

- Standard mechanisms for Zynq host-fabric communication (AXI GP, AXI HP, AXI ACP, AXI streaming with DMA).
- Hardware emulators for storage devices generally (USB mass storage emulation, SATA emulation, etc.).
- Block-device backends in virtualisation (loop devices, network block devices, etc.).
- Filesystem-as-interface patterns (FUSE, /proc, /sys in Linux).

The specific contribution of Virtual SD I/O is the application of SD card emulation to the host-fabric communication problem in FPGA SoCs, with the streaming-interface-backed configuration providing high-bandwidth data transfer through familiar interfaces on both sides.

## Remarks

The idea applies to any FPGA SoC family where the hard SD controller can be routed into the programmable logic fabric. AMD Zynq 7000 is one explicit example. Other families with similar architectural flexibility (other AMD FPGA SoCs, Intel Agilex SoC, Microchip PolarFire SoC, others) may support the same approach with appropriate adaptation.

Implementation of the fabric-side SD card emulator is the substantive engineering work. The emulator must respond to the SD protocol commands the host issues, manage the protocol state correctly, and bridge to whatever backend (BRAM, external memory, AXI streaming) the configuration uses. SD card protocol implementations exist as open-source IP cores and as proprietary commercial cores; either can be adapted as the basis for a Virtual SD I/O emulator.

The configurations above are the obvious initial uses. Other configurations are possible — partial-storage hybrids, multi-channel streaming, dynamically reconfigurable mappings, and so on. The protocol is flexible because the SD card abstraction is a general block-device interface that can stand in for many backend behaviours.
