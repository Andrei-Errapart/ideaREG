## Idea

### Title

**ASIO** (A Simple I/O) — memory-mapped GPIO IP core with implicit direction control through access type.

### Description

ASIO is an IP core that provides a memory-mapped GPIO interface for FPGA designs. The distinctive design feature is that **the direction of each pin is determined implicitly by the access type rather than by a separate direction control register**.

**The register map.** Each pin is associated with one address. For pin N, the address is `base_address + N * 4` (one word per pin, with the *4 stride being the standard 32-bit-aligned word spacing of most memory-mapped buses).

**The access behaviour.** Two operations:

- **Write to pin N's address.** The pin becomes an output. The LSB of the written value determines whether the output is driven high (write 1) or low (write 0). The pin remains an output until a subsequent read or write changes its state.

- **Read from pin N's address.** The pin becomes an input. The current value at the pin is sampled and returned in the LSB of the read data. The pin remains an input until a subsequent write changes it to an output.

This eliminates the conventional separate direction register that other GPIO IP cores use. Software using ASIO simply reads from a pin to use it as an input and writes to a pin to use it as an output. The mental model maps directly to what the software wants to do.

### Why this idea matters

Conventional GPIO IP cores have three registers (or three bitfields) per pin: direction control, output value, input value. Software using these cores must configure the direction register before each direction change, then access the output or input value register depending on the direction. This is correct but verbose, and the register map is harder to remember than it could be.

ASIO collapses this to one address per pin, with the access type encoding the direction. The result:

- **Simpler software.** Setting a pin high as output is one write. Reading a pin as input is one read. No direction configuration step.
- **Easier to document and remember.** Pin N is at offset N*4. The access type is the operation.
- **Lower implementation cost in the FPGA.** One register per pin instead of three.

The trade-off is that there is no way to read the current output value of a pin without converting it to an input. For use cases where this matters, ASIO is not the right choice. For the many use cases where it does not matter (and most GPIO use cases do not), ASIO is simpler than the conventional design.

### Domain or field

FPGA IP core design, memory-mapped peripherals, GPIO controllers, embedded system interfaces.

### References to related ideas

- **AMD/Xilinx AXI GPIO** and similar GPIO IP cores from other vendors — the conventional three-register-per-pin design that ASIO simplifies.
- **Memory-mapped I/O patterns in general** — the use of address-mapped registers for hardware control is standard practice in embedded systems; ASIO applies this pattern with the specific access-type-as-direction-control choice.
- **MMIO conventions in CPU architectures** — some CPU instruction sets distinguish read-with-side-effect from read-without-side-effect at the instruction level. ASIO's behaviour is similar in spirit but implemented at the IP core level.

The specific contribution of ASIO is the access-type-as-direction-control encoding, which removes the direction register that conventional GPIO designs use and produces a simpler software model and a smaller implementation.

## Remarks

ASIO is intended as an IP core for FPGA designs that need GPIO functionality with minimal software and hardware overhead. The design is straightforward to implement on any memory-mapped bus interface (AXI-Lite is the most common; Avalon and Wishbone are also plausible).

The trade-off — that pin output values cannot be read back without converting the pin to an input — is a real limitation. Designs that need to verify their output settings through readback would need a different GPIO core or an additional readback mechanism. For the majority of GPIO use cases, where software either writes outputs or reads inputs but does not read back outputs, ASIO is appropriate.

The "A Simple I/O" naming reflects the design philosophy. The core is intentionally minimal. There are no interrupt outputs, no edge-detection logic, no pull-up/pull-down configuration registers, no glitch filters. These features can be added as separate IP cores or external logic when needed. ASIO does one thing — the simplest possible memory-mapped GPIO — and does it with minimal complexity.
